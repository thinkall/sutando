# Workspace design — the 3-space model

**Status:** RFC. Open alignment decisions appear inline as "Decisions" at the bottom; reactions welcome in the matching tracking issue.

## Why this doc exists

Through 2026-05-18 a workspace-vs-repo bug class surfaced ~28 sites across Python, TypeScript, and Swift where writers and readers disagreed about whether a path resolved against the git checkout or against per-user runtime state. 16 PRs closed the divergence (#815, #821, #832, #833, #836, #837, #838, #842, #843, #844, #846, #848, #849, #855, plus PR #837 on the Swift side). The runtime contract is now end-to-end consistent — but the *vocabulary* wasn't: `REPO_DIR`, `WORKSPACE_DIR`, `PRIVATE_DIR`, sometimes `process.cwd()`, sometimes `import.meta.url`-relative — no single mental model that says which is which.

**Three spaces, not three env vars.** Each space has a purpose, a sync rule, and a resolver. Env vars are implementation; the spaces are the design.

## Quick decision: which space?

When adding code that reads or writes a file, walk this list top-to-bottom and stop at the first match:

1. **Does it ship with the code (same on every install)?** → **Code**. Examples: a new `.py` script, a `.yaml` config in `skills/`, a markdown doc.
2. **Does the user read or directly edit it?** → **Memory** (Shared if same across all their Macs, Per-machine if each Mac has its own). Examples: notes, build_log, agent memory entries, stand identity.
3. **Is it generated by running code, rotates, and has no value across machines?** → **State**. Examples: logs, PID files, task queue, results, briefings, dynamic-content.json.

If you can't place a path on this flowchart in 5 seconds, the path probably wants splitting (e.g. a config file with both shared defaults + per-machine overrides should live in two places).

## TL;DR

| Space | Purpose | Lives at | Sync | Lifecycle |
|---|---|---|---|---|
| **Code** | Source of truth for behavior | `$SUTANDO_REPO_DIR` (git checkout) | Public git (`sonichi/sutando`) | Persistent; updated by `git pull` |
| **State** | Per-user runtime, mostly machine-local | `$SUTANDO_WORKSPACE` (default `~/.sutando/workspace/`) | None — per-machine | Ephemeral; rebuildable; rotates |
| **Memory** | User-content the user reads + edits | `$SUTANDO_PRIVATE_DIR` (proposed rename → `SUTANDO_MEMORY_DIR`) | Private git per fleet (`<user>-sutando-memory.git`) | Persistent; user-authored |

Within Memory, two flavors:
- **Shared** (top-level): one copy across the fleet. Examples: `notes/`, `build_log.md`, `memory/*.md`.
- **Per-machine** (`machine-<host>/` subdir): each Mac has its own copy. Examples: `stand-identity.json`, `pending-questions.md`.

## The 3 spaces in detail

### Code

The git repo. Source of truth for behavior. Owned by version control.

**Resolvers:**
- Python: `Path(__file__).resolve().parent.parent` for src-adjacent paths; `$SUTANDO_REPO_DIR` if explicitly needed (post-#831 for scripts like `sync-memory.sh`).
- TypeScript: `dirname(dirname(fileURLToPath(import.meta.url)))` (post-#848 pattern in `src/inline-tools.ts`; matches `src/web-client.ts:27` post-#821).
- Swift: `AppDelegate.repoRoot` (post-#837).

**Lives at:** wherever the user cloned. `~/Desktop/sutando`, `~/Documents/github/sutando`, etc.

**Sync model:** standard public git. `git pull` keeps in sync.

**What goes here:** `src/`, `skills/`, `scripts/`, `tests/`, `docs/`, `CLAUDE.md`, `README.md`, `package.json`, `.env.example`. Anything that should be the same across every install.

**What does NOT go here:** anything per-user. `tasks/`, `results/`, `state/`, `logs/`, `.env`, `conversation.log`, the actual `notes/` and `build_log.md` — none of these are code.

### State

Per-user runtime state. Machine-local by default. Rebuildable.

**Resolvers:**
- Python: `from workspace_default import resolve_workspace` (post-#762).
- TypeScript: `import { resolveWorkspace } from './workspace_default.js'` (post-#821).
- Swift: `AppDelegate.workspace` (post-#837).

**Lives at:** `$SUTANDO_WORKSPACE` or `~/.sutando/workspace/` (default).

**Sync model:** none by default. Each Mac has its own State. If a user wants to inspect across machines they'd do it manually.

**What goes here:** `tasks/`, `results/`, `state/`, `logs/`, `data/`, `conversation.log`, `core-status.json`, `voice-state.json`, `dynamic-content.json`, `contextual-chips.json`, `.voice-agent.pid`. Anything that:
- Is generated by running code (not user-authored).
- Rotates or can be re-derived.
- Has no value across machines (logs from MacBook aren't useful on Mac Mini).

**Lifecycle:** ephemeral. `rm -rf <workspace>` should be survivable — startup re-creates whatever's needed.

### Memory

User-content. The user reads it, edits it directly, expects it to follow them. Synced across the fleet.

**Resolvers:**
- Python: `from util_paths import personalPath, sharedPersonalPath`.
- TypeScript: `import { personalPath, sharedPersonalPath } from './util_paths.js'`.

**Lives at:** `$SUTANDO_PRIVATE_DIR` (e.g. `~/.sutando-memory/`). Falls back to State when unset — this is **intentional onboarding friction reduction**, not a bug. Fresh installs work without setting `SUTANDO_PRIVATE_DIR`; the user can opt into a separate Memory location later. The lifecycle distinction is documented but not enforced.

**Sync model:** private git remote (`<user>-sutando-memory.git`). `scripts/sync-memory.sh` is the canonical sync.

**Two flavors inside Memory:**

```
$SUTANDO_PRIVATE_DIR/
├── notes/                  # SHARED — anything I save once, visible everywhere
├── build_log.md            # SHARED — project log, append-only
├── memory/                 # SHARED — agent's auto-memory index + entries
├── voice-contexts/         # SHARED — per-context speech profiles
└── machine-<host>/         # PER-MACHINE — host-specific identity + state
    ├── stand-identity.json # this Mac's Stand name
    └── pending-questions.md # this session's open decisions
```

`personalPath(filename)` resolves to `machine-<host>/<filename>` if it exists, else falls back to top-level. `sharedPersonalPath(filename)` always resolves to top-level.

**Lifecycle:** persistent. Loss = real data loss. Backups via the private git remote are the recovery story.

## Why this framing now

The 2026-05-18 audit closed the **runtime-data divergence** bug class — Python writers had migrated to `$SUTANDO_WORKSPACE` while some TS/Swift readers still resolved to repo paths, silently splitting state on `$SUTANDO_WORKSPACE`-set hosts. The audit was painful precisely because the **vocabulary** was inconsistent: `REPO_DIR`, `WORKSPACE_DIR`, `PRIVATE_DIR`, occasionally `process.cwd()`, occasionally `import.meta.url`-relative — no single mental model that says which is which.

Three spaces, not three env vars. Each space has a purpose, a sync rule, and a resolver. Env vars are implementation; the spaces are the design.

## Multi-machine model

A user can have multiple Macs (`macbook`, `mini`). They want some content shared (notes, memory) and others isolated (logs, conversation history).

This model handles it in two ways:

1. **State stays per-machine.** MacBook's logs don't belong on Mac Mini. Voice-agent's PID file is meaningless across hosts. No sync needed.

2. **Memory has Shared + Per-machine subdivision** inside the private git repo. The `personalPath` vs `sharedPersonalPath` helpers in `src/util_paths.{py,ts}` already implement this; the design just names what it is.

## Test isolation discipline

Tests that write runtime state MUST use a per-process temp workspace, never `REPO_ROOT`. Pattern established in #849:

```ts
import { mkdtempSync, rmSync } from 'node:fs';
import { tmpdir } from 'node:os';
import { join } from 'node:path';

const TEMP_WORKSPACE = mkdtempSync(join(tmpdir(), 'sutando-test-<name>-'));
process.env.SUTANDO_WORKSPACE = TEMP_WORKSPACE;  // BEFORE any import that captures WORKSPACE_DIR
// ... test body writes/reads via WORKSPACE_DIR-relative paths
// after():
rmSync(TEMP_WORKSPACE, { recursive: true, force: true });
```

A spawned subprocess receives `SUTANDO_WORKSPACE: TEMP_WORKSPACE` in its env so `resolveWorkspace()` matches. This eliminates the cross-test-file race class that produced flaky CI for issues #800 / #840.

## Migration cost (if accepted)

Low — mostly documentation. The runtime contract is already correct end-to-end.

- One semantic change worth considering: rename `SUTANDO_PRIVATE_DIR` → `SUTANDO_MEMORY_DIR` (with legacy alias for one release). Touches `scripts/sync-memory.sh` + `src/util_paths.{py,ts}` only — ~5 files.
- Bigger work: doc + vocabulary shift in CLAUDE.md and onboarding materials.

No re-audit needed.

## A note on IDE / editor config

Files like `.cursorrules`, `.claude/` (project-local Claude Code config), and `.vscode/` are sometimes per-user and sometimes shared. The 3-space model doesn't have a clean home for them:

- If checked in (e.g. `.vscode/extensions.json` recommendations), they're **Code** — shared with everyone who clones.
- If per-user (e.g. `.vscode/settings.json` with personal font preferences), they're effectively **Memory** but live at the repo root for tool-discovery reasons.
- The practical answer: `.gitignore` the per-user variants, check in the shared ones. Same convention every IDE-aware repo uses. The 3-space model doesn't change this — it's an existing convention that maps to Code (shared) or stays out (per-user, untracked).

Worth flagging because new contributors sometimes ask "should `.cursorrules` go in Memory?" — the answer is no, it stays in Code-or-untracked per IDE convention.

## Host CLI dependency surface

Sutando rides on a host CLI tool (today: Claude Code) for a handful of agent-state surfaces — agent memory, skill discovery, bridge tokens, slash commands — that live inside the host CLI's user-home directory rather than under any Sutando-owned space. See **[`docs/host-cli-bindings.md`](host-cli-bindings.md)** for the surface taxonomy, the per-surface re-bind cost on a host-CLI swap, and the canonical helper (`claude_home_path()` / `claudeHomePath()`) you should call when new code touches the host CLI's home.

Short version for this RFC: the dependency surface exists, it's intentionally countable through a single greppable helper, and specific path strings stay in the workspace-private engineering note (publishing them in public docs invites bit-rot when the host CLI's conventions change).

## Decisions

The RFC opens 6 decisions, all **ratified at PR #858 merge** (3 endorsements, no objections). Items with implementation work are tracked in separate issues — linked below. The summaries here are the doc-reference; the issues are where impl progress lives.

### 1. Naming the spaces

**Proposed:** Code / State / Memory.

**Alternatives considered:** Source / Runtime / Library; Repo / Workspace / Notes.

**Rationale for proposal:**
- "Code" is unambiguous (vs "Repo," which could mean the *private* memory git repo).
- "State" captures ephemeral + rebuildable.
- "Memory" captures the user-edit + agent-recall double-use.

**Status:** Ratified — vocabulary lands with this PR's merge. No follow-up issue needed.

### 2. Env-var rename: `SUTANDO_PRIVATE_DIR` → `SUTANDO_MEMORY_DIR`

**Proposed:** Rename, with one-release alias period for backwards compatibility. Log a deprecation warning **every time** the legacy var is read, not just once at startup (cron environments miss startup-only warnings).

**Rationale:** "Private" reads like "secrets dir" but the contents are user-content. The mismatch causes onboarding friction; every new contributor wonders what "private" means.

**Migration scope:** ~5 files (`util_paths.{py,ts}`, `scripts/sync-memory.sh`, possibly bundle resolver). Single-release deprecation.

**Status:** Ratified. Implementation tracked in [#870](https://github.com/sonichi/sutando/issues/870).

### 3. Host segmentation in Memory

**Proposed (pragmatic middle):** Keep `hostname().split('.')[0]` as the default, **add `SUTANDO_HOST_LABEL` env override** that takes precedence.

**Alternatives considered:** UUID per install (stable but opaque); short user-chosen name only (requires onboarding prompt).

**Rationale:** Zero migration cost for existing installs. Users who care about stable identity (rename-Mac-in-Settings, multi-install on one machine) opt in by setting the label. UUID is cleanest theoretically but opaque-in-folder-tree adds friction when the user inspects the private repo.

**Status:** Ratified. Implementation tracked in [#871](https://github.com/sonichi/sutando/issues/871).

### 4. Should State *ever* sync?

**Proposed:** Default no. Add an **opt-in `.sync-allowlist` file** at the workspace root that lists specific files to include in memory-sync. Most users never touch it; advanced users get the path without weakening the default.

**Rationale:** Logs are huge and machine-meaningless — sync would be wasteful. But selective aggregation (e.g. `call-metrics.jsonl` across Macs for analytics) is a real use case. Allowlist gives the escape hatch without making "should this sync?" a per-file question for every new path.

**Open sub-thread:** The shared-task-queue case (Qingyun raised 2026-05-18T20:09Z) is the strongest pull toward syncing more of State. Possible refinement: introduce a `state/fleet/` subdir inside `$SUTANDO_WORKSPACE` for paths that explicitly want cross-machine visibility (task queue, results), with a coordination protocol (claim files via atomic `mv`). This is option (B) from my proposal; the allowlist (this decision) is option (B)-lite.

**Status:** Ratified default-no + allowlist. The `state/fleet/` sub-axis is deferred to a separate design pass. Both tracked in [#872](https://github.com/sonichi/sutando/issues/872).

### 5. Secrets (`.env` and similar)

**Proposed:** Stay where they are (Code-adjacent `.env`, gitignored). Do not promote secrets to a 4th space.

**Alternatives considered:** 4th space ("Secrets"); fold into Memory with `.gitignore` discipline.

**Rationale:**
- Most users want their API keys to follow them across Macs, but the right answer there is a **separate "personal sync via 1Password CLI / system keychain" pattern** — not folding into Memory.
- Adding a 4th space increases the model's surface area without a clear win.
- Memory-with-gitignore is risky: a Memory bug that leaks a notes/ file is annoying; one that leaks an API key is a security incident.

**Status:** Ratified — secrets stay at `.env`. The "keys across Macs" UX problem is deferred to a separate workstream (1Password CLI / system Keychain). No follow-up issue for this PR; the cross-Mac-keys workstream will get its own issue when scoped.

### 6. Sutando.app bundle reconciliation

**Proposed:** Track as an independent reconciliation PR; the 3-space model doesn't change its resolution. The 9 Swift files in the deployed app bundle that are missing from `src/Sutando/` (pending decision from 2026-05-13) are a Code-space gap regardless of vocabulary.

**Rationale:** Workspace design is orthogonal to "which Swift files are in git." Treating them as the same conversation slows both.

**Status:** Ratified — tracked separately under the pre-existing 2026-05-13 bundle-reconciliation issue.
